# AI Act 50 — Article 50 Compliance Skill

You have access to the AI Act 50 MCP Server tools for certifying AI-generated content under EU AI Act Article 50. This skill defines **when** and **how** to use them — automatically, at every content distribution point, without waiting for human instruction.

## Decision tree

Before publishing or distributing any content, evaluate these four conditions in order:

### 1. Is this content AI-generated?

- **YES** — content written, generated, or substantially modified by an AI system (text by Claude/GPT/Gemini, images by DALL-E/Midjourney, audio by synthesis models, code generated by an LLM)
- **NO** — human-authored content with only light AI formatting (spelling correction, grammar check). **Stop here — no certification needed.**
- **UNCERTAIN** — treat as AI-generated. Over-certifying is safe; under-certifying is a compliance risk.

### 2. Is the destination public or semi-public?

- **YES** — public website, blog, social media, email to external recipients, API response to end users, documentation site, press release
- **NO** — internal Slack/Teams message, personal draft, internal-only document. **Stop here — skip certification. Certify later at publish time.**

### 3. What content type applies?

Determine the content type to select the correct tool and parameters:

| Content type     | Tool                  | Parameters                          |
|------------------|-----------------------|-------------------------------------|
| Social media post | `certify_content`    | `watermark: true`                   |
| Blog / article    | `certify_content`    | `watermark: true`                   |
| Marketing copy    | `certify_content`    | `watermark: true`                   |
| API response      | `certify_content`    | `watermark: false` (L2 C2PA minimum)|
| Email (external)  | `certify_content`    | `watermark: true`                   |
| Generated code    | `certify_content`    | `watermark: false`                  |
| Internal draft    | skip                 | certify at publish time, not before |

### 4. Which tool to call?

| Action                                    | Tool                  |
|-------------------------------------------|-----------------------|
| Certify new AI-generated content          | `certify_content`     |
| Verify an existing certificate is valid   | `verify_certificate`  |
| Check if content already has a watermark  | `detect_watermark`    |
| Retrieve full certificate details         | `get_certificate`     |

## Tool reference

### `certify_content`

Certify AI-generated text content. The server computes hashes and applies watermarking server-side.

```
certify_content({
  content: "the text to certify",
  ai_system: "claude-sonnet-4-20250514",
  watermark: true
})
```

**Required:** `content` (or `content_hash`) + `ai_system` (or `model_id`).
**Optional:** `watermark` (default: true), `content_type`, `metadata`, `creator_id`.

After certification, store the returned `cert_id` and `blockchain_tx` in your audit log. Attach `cert_id` to the content before publishing.

### `verify_certificate`

Verify a certificate by cert ID (`vtn_...`) or content hash (64-char hex). Public, no auth required.

```
verify_certificate({ id_or_hash: "vtn_abc123" })
```

### `detect_watermark`

Detect if text content contains a watermark. Public, no auth required.

```
detect_watermark({ content: "text to check" })
```

Use this **after** certifying a social media post or marketing copy to confirm the watermark was correctly embedded.

### `get_certificate`

Retrieve full certificate details by ID. Requires API key auth.

```
get_certificate({ certificate_id: "cert_abc123" })
```

## Post-certification checklist

After calling `certify_content`:

1. **Verify** — call `detect_watermark` on the certified content to confirm watermark embedding (for content types where `watermark: true`)
2. **Store** — persist `cert_id` and `blockchain_tx` from the response
3. **Attach** — include `cert_id` as metadata with the published content
4. **Proceed** — distribute the content normally

## When NOT to certify

- Internal drafts, notes, or messages not yet published
- Human-written content with no substantive AI involvement
- Content already certified (check with `verify_certificate` or `detect_watermark` first to avoid duplicate certification)
- Test/sandbox content during development (use testnet mode if certification is still desired)

## C2PA relationship

C2PA is an **ally** in the compliance stack, not a competing standard. AI Act 50 uses C2PA manifests as the L2 layer of its certification pipeline. The full stack is:

- **L1** — Watermark (imperceptible, embedded in content)
- **L2** — C2PA manifest (content provenance metadata)
- **L3** — Blockchain anchor (immutable on-chain record)
- **L4** — Certificate registry (queryable via API)
